Sistema de Streaming

Cliente / Servidor com estruturas de dados implementadas à mão

Java puro, sem frameworks  ·  Estruturas de Dados II

Árvore Splay Compressão Huffman Cache AVL Tabela Hash Tabela de Tradução

O que o projeto faz

  • Simula um streaming cliente↔servidor com catálogo de 1000 filmes reais (IMDB Top 1000).
  • O usuário pesquisa pelo nome do filme; internamente tudo é resolvido por ID.
  • Multiusuário: cada um tem cache e recomendações próprias; o servidor é compartilhado.
  • Toda a comunicação trafega comprimida com Huffman — nenhuma ida ao servidor escapa do canal.

Tudo construído sem bibliotecas de estrutura de dados: apenas arrays, nós e referências.

1. Tabela Hash (servidor)

  • Banco do servidor: 1000 filmes em 211 posições.
  • Índice: Math.floorMod(id, capacidade).
  • Colisões por encadeamento exterior (lista ligada por balde) — ~5 filmes por balde.
  • A busca conta as comparações percorridas na cadeia.
public BuscaFilmeResultado buscarPorId(int id) {
  int indice = calcularIndice(id);
  int comparacoes = 0;
  EntradaHash e = tabela[indice];
  while (e != null) {
    comparacoes++;
    if (e.chave == id)
      return new BuscaFilmeResultado(e.filme, comparacoes);
    e = e.proxima;          // anda na corrente
  }
  return new BuscaFilmeResultado(null, comparacoes);
}

2. Tabela de Tradução (cliente)

  • Hash própria com chave no nome normalizado e valor no ID.
  • Montada no início: o cliente recebe o índice nome→id do servidor comprimido e popula a tabela.
  • Permite pesquisar por nome sem mudar a hash do servidor nem o cache AVL (que continuam por ID).
  • Tolerante a maiúsculas/minúsculas (a chave é sempre normalizada).

É o "tradutor" que conecta a pesquisa humana (nome) ao fluxo interno (ID).

3. Cache AVL (cliente)

  • Árvore balanceada ordenada por ID → busca O(log n).
  • Capacidade 50 (menor que o catálogo, de propósito).
  • Eviction FIFO: cada nó guarda a ordemEntrada; ao lotar, sai o que entrou primeiro.
  • HIT no cache = resposta local, sem tocar a rede.
private class NoAVL {
  K chave; V valor;
  long ordemEntrada;   // define quem sai (FIFO)
  int altura;          // mantém o balanceamento
  NoAVL esquerda, direita;
}

Balanceamento por altura + rotações; remoção escolhe o nó de menor ordemEntrada.

4. Árvore Splay — recomendações

O destaque do projeto

Genérica, usada nos dois lados:

  • Splay do cliente → cada filme consultado sobe; o topo é a recomendação pessoal.
  • Splay do servidor → cada filme servido a qualquer um sobe; o topo é a recomendação global.

A ideia central do "splay"

  • A cada acesso (busca ou inserção), o nó acessado é levado até a raiz por rotações.
  • Consequência: o que é acessado com frequência/recência fica perto da raiz — barato de achar de novo.
  • Isso vira recomendação natural: o topo da árvore = o que está em alta.
acesso a D:           splay(D)
      B                    D
     / \                  / \
    A   E      ──►        B   E
       / \              / \    \
      D   F            A   C    F
     /
    C

Splay top-down clássico de Sleator & Tarjan (sem recursão por nó, reorganiza num passe só).

Splay — buscar marca recência

public BuscaCacheResultado<V> buscar(K chave) {
  if (raiz == null) return new BuscaCacheResultado<>(null, 0);
  int comparacoes = contarComparacoes(chave);
  raiz = splay(raiz, chave);                 // traz a chave p/ a raiz
  if (raiz.chave.compareTo(chave) == 0) {
    raiz.ultimoAcesso = ++relogio;           // carimba o instante
    return new BuscaCacheResultado<>(raiz.valor, comparacoes);
  }
  return new BuscaCacheResultado<>(null, comparacoes);
}

Cada nó guarda ultimoAcesso (um relógio lógico). Esse carimbo define recomendação e eviction.

Splay — capacidade 50 e eviction LRU

Ao passar de 50 filmes, remove o de acesso mais antigo (LRU):

if (capacidade > 0 && tamanho > capacidade)
    removerMenosRecente();
// remover = splay da chave p/ raiz
//           + junta as subárvores

Recomendar = ordenar por ultimoAcesso desc:

public List<V> recomendar(int qtd) {
  List<NoSplay> nos = new ArrayList<>();
  coletar(raiz, nos);
  nos.sort(comparingLong(
     n -> n.ultimoAcesso).reversed());
  // devolve os 'qtd' mais recentes
}

Por isso usuários diferentes veem recomendações pessoais diferentes, mas a do servidor reflete todos.

5. Compressão Huffman — o canal

Toda mensagem entre cliente e servidor é comprimida antes de "trafegar".

Caracteres frequentes ganham códigos curtos; raros, códigos longos.

Huffman em 4 passos

  1. Frequências — conta cada caractere do texto (HashMap).
  2. Árvore — cada caractere vira folha; combina sempre os 2 nós de menor frequência até sobrar 1.
  3. Códigos — caminho raiz→folha (0 esquerda, 1 direita).
  4. Empacota — concatena os bits e fecha em byte[].
while (floresta.size() > 1) {
  NoHuffman menor       = removerMenor(floresta);
  NoHuffman segundoMenor = removerMenor(floresta);
  floresta.add(new NoHuffman(menor, segundoMenor)); // pai = soma das freq.
}
return floresta.get(0);   // raiz da árvore de Huffman

Decodificação e o Pacote

Percorre a árvore bit a bit até cair numa folha:

for (int i = 0; i < totalBits; i++) {
  int bit = (dados[i/8] >> (7-(i%8))) & 1;
  atual = bit==0 ? atual.esquerda
                 : atual.direita;
  if (atual.folha()) {
    texto.append(atual.caractere);
    atual = raiz;
  }
}

O Pacote carrega tudo p/ decodificar:

  • byte[] dados — bits compactados
  • totalBits — bits válidos (ignora padding)
  • arvore — o "dicionário"
  • bytesOriginais — p/ calcular a taxa

Caso de borda: texto de 1 símbolo só → árvore vira uma folha, código "0".

O Canal: Huffman + estatísticas

public Pacote comprimir(String texto) {
  Pacote p = Huffman.comprimir(texto);
  mensagensEnviadas++;                       // contabilidade
  bytesOriginais   += p.getBytesOriginais();
  bytesComprimidos += p.getBytesComprimidos();
  return p;
}
public String descomprimir(Pacote p) { return Huffman.descomprimir(p); }

Canal único e compartilhado. Cada requisição e cada resposta é comprimida separadamente.
Saída típica: reducao=51.28% numa busca.

O fluxo principal

Da inicialização até uma busca atravessar o canal

1 · Inicialização (Main.main)

main()
 ├─ new CanalComunicacao()              cria o canal (compartilhado)
 ├─ new ServidorStreaming(211, canal)   hash de 211 posições
 ├─ servidor.carregarBaseDeArquivo()    lê filmes.csv -> popula a hash
 ├─ valida total == 1000 filmes
 ├─ imprime cabeçalho
 └─ executarLogin(servidor, canal, ...) entra no laço de login

Um único canal e um único servidor são criados e passados adiante. O servidor já nasce com toda a base na TabelaHashFilmes.

2 · Login multiusuário (executarLogin)

loop tela de login:
   lê nome do usuário
   ├─ "0"  -> encerra o programa
   ├─ nome novo:
   │     new ClienteStreaming(nome, servidor, canal, 50)
   │     cliente.sincronizarTabelaTraducao()  <-- 1ª comunicação comprimida!
   │     guarda no mapa de usuários
   └─ executarSessao(...)                      entra no menu

A sincronizarTabelaTraducao() pede GET_INDICE, recebe o índice nome→id comprimido e monta a tradução local.
Cada usuário tem cache AVL e Splay próprios; o servidor é compartilhado.

3 · Menu da sessão (executarSessao)

Laço do/while que lê a opção e despacha. Ex.: opção 3 →

buscar(...)  ->  cliente.consultarPorNome(nome)

Outras opções: listar categorias, 20 consultas aleatórias (popula recomendações), mostrar cache, recomendações, estatísticas Huffman, logout.

4 · O coração: uma busca por nome

consultarPorNome(nome)                      [ClienteStreaming:99]
 ├─ traducao.traduzir(nome) -> id            tabela local (SEM rede)
 │    nome inexistente? imprime e retorna
 ├─ snapshotCanal()                          foto dos contadores ANTES
 ├─ consultarFilme(id)
 │    ├─ cache.buscar(id) (AVL)
 │    │    HIT -> registra recom. (Splay local) e retorna  ── SEM REDE
 │    │    MISS v
 │    └─ enviar("GET_ID|")               <---- canal comprimido
 │         ├─ canal.comprimir(texto) -> Pacote   (Huffman + stats)
 │         ├─ servidor.atender(Pacote)       [ServidorStreaming:89]
 │         │    ├─ canal.descomprimir -> texto
 │         │    ├─ processar: hash.buscarPorId(id)
 │         │    │    + registrarRecomendacao (Splay GLOBAL)
 │         │    └─ canal.comprimir(resposta) -> Pacote
 │         └─ canal.descomprimir(Pacote) -> texto da resposta
 │         cache.inserir(id, filme)   |   registrarRecomendacao (Splay local)
 └─ imprimirCompressaoDaPesquisa(antes)      diferença = custo desta busca

Pontos-chave do fluxo

  • Ponto único de comunicação: cliente sempre via enviar (comprime→envia→descomprime); servidor sempre via atender (descomprime→processa→comprime).
  • Duas estruturas evitam/tocam a rede:
    • Tradução (nome→id) e cache AVL ficam no cliente → HIT não gera comunicação.
    • Só no MISS o pacote viaja e bate na TabelaHashFilmes.
  • Duas Splay, dois pontos de alimentação: cliente (gosto pessoal) e servidor (popularidade global).
  • Estatística por snapshot: fotografa os contadores antes e subtrai depois → isola o custo daquela busca; opção 10 mostra o acumulado da sessão.

Resumo

  • Splay transforma recência em recomendação — sem estrutura extra.
  • Huffman comprime 100% do tráfego; integridade preservada (comparações voltam intactas).
  • Hash + AVL + Tradução sustentam busca por nome resolvida por ID, com cache local.
  • Tudo em Java puro, implementado manualmente.
